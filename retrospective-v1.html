4. Best Practices — What the Code Actually Shows

Adapter Pattern for AI Extraction Excellent

src/lib/extraction/adapter.ts (19 lines) — switches between claudeExtract and mockExtract based on ANTHROPIC_API_KEY env var. This is a textbook pluggable adapter that enables development without API costs, demo mode, and future provider swaps.

Why it works: The type signature ExtractionAdapter = (fileName, fileBuffer, projectId?) => Promise<ExtractionResult> creates a clean contract. The mock returns realistic data (732-drop Highlark fixture), so the entire BOM/labor pipeline can be tested without API calls.

Real-World Validation Testing Excellent

src/lib/calculators/__tests__/highlark-validation.test.ts (127 lines) — validates the entire estimation pipeline against a known real bid ($304,955 / 1,748 hours). Each assertion has a ±5% tolerance band against the actual Highlark Hotel estimate.

Why it works: This is not a typical unit test — it is a calibration harness. When pricing constants change, this test fails before bad estimates reach clients. The fixture (highlark-hotel.ts) contains 732 synthetic drops matching the real project scope.

Structured Observability Good

src/lib/logger.ts (60 lines) — structured JSON logging with module scoping, timing helpers, and request ID correlation (injected by proxy.ts). Usage logs are also persisted to PostgreSQL (UsageLog model) for API cost tracking.

Improvement: Currently uses console.log output. For Phase 2, consider Vercel's built-in log drains or a structured logging integration (Datadog, Sentry) via the Marketplace.

Zod Schema as API Contract Excellent

Claude's extraction output is validated against a strict Zod schema (extractionSchema in claude-adapter.ts:16-50) before entering the pipeline. Drop types, cable types, and conduit types are constrained to enums. This catches LLM hallucination at the boundary.

Why it works: LLMs can return unexpected formats. The .safeParse() call converts a raw JSON blob into a type-safe ExtractionResult or throws with detailed issue descriptions. Markdown fence stripping handles the common ```json wrapping.

Next.js 16 proxy.ts Good

src/proxy.ts (14 lines) — correctly uses the Next.js 16 proxy.ts convention (not deprecated middleware.ts). Injects x-request-id headers for request tracing.

Note: Correctly placed in src/ (the project uses --src-dir). Follows Vercel best practices for middleware → proxy migration.

API Error Handling Good

claude-adapter.ts:219-236 — catches Anthropic-specific exceptions (AuthenticationError, BadRequestError, RateLimitError) and converts them to user-friendly messages. Credit balance errors get a specific redirect message.

Improvement: The catch block at line 214 (usage log write failure) silently swallows errors. Consider logging the actual error for debugging.

7. Key Findings & Recommendations

What Worked Best

Superpowers design-then-plan-then-execute The calibration feature is the project's crown jewel (±5% accuracy). It's also the only feature that went through the full brainstorm → design spec → implementation plan → execution workflow. Correlation is clear: structured planning produced the highest-quality output.
Adapter pattern for AI The extraction adapter enabled development without API costs, demo mode for client presentations, and future provider flexibility. This 19-line file is the most valuable abstraction in the codebase.
Real-world calibration tests Testing against a known $305K bid (not synthetic data) catches pricing regressions that would cost the client money. This is a domain-specific best practice worth replicating.
PRD from code Generating the PRD from the existing codebase (not from scratch) ensured the Phase 2 proposal was grounded in real capabilities. The simulation report caught architectural risks early.

What Could Be Improved

Day 1 had no planning phase 15 commits in one day built the entire foundation — impressive velocity, but no design spec or plan artifact. The Prisma Decimal serialization bug (8e2c85c) and category name case mismatch (73ee79e) were avoidable with a superpowers:brainstorming pass first.
Tests written after implementation Commit timestamps show implementation first, tests second. The superpowers:test-driven-development skill wasn't used. For a tool where pricing accuracy = money, TDD would catch bugs before they compound.
Direct Anthropic SDK vs AI Gateway Using @anthropic-ai/sdk directly locks the project to one provider with manual API key management. AI Gateway + OIDC would give automatic cost tracking, failover, and no secrets to manage.
Refine plugin overhead 14 commits (21% of all commits) went to the refine experiment on a .website/ directory. While the scoring methodology is interesting, the time could have been spent on core bid tool features.
No CLAUDE.md The project has no CLAUDE.md file. This means every new Claude Code session starts without project context, conventions, or constraints. Creating one would encode the domain knowledge (BICSI standards, NEC compliance, BOM categories) that currently lives only in code comments.

Phase 2 Recommendations

Add CLAUDE.md Encode domain conventions (BICSI labor categories, NEC conduit fill rules, drop type taxonomy), test commands, and the Highlark calibration target. This becomes the project's institutional memory.
Adopt TDD via Superpowers Use superpowers:test-driven-development for all Phase 2 features, especially BOM persistence (PREREQ-001) and multi-building aggregation (F-104) where pricing bugs cascade.
Migrate to AI Gateway Replace @anthropic-ai/sdk with AI SDK + AI Gateway when adding the production extraction adapter (F-120). This enables cost tracking per-project, provider failover, and eliminates API key management.
Use verification-before-completion Require superpowers:verification-before-completion before every commit that touches calculators. The ±5% Highlark test must pass before any pricing change ships.
Use Clerk via Marketplace For Theme 3 (auth), use Clerk instead of Auth.js. The PRD already flagged Auth.js v5 compatibility risk. Clerk auto-provisions env vars via Vercel Marketplace.
