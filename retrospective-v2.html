Planning vs. Implementation Sequence

Phase 1 (Mar 2–4): Implementation-first. No design doc preceded the initial build. 22 feature commits in 3 days, all driven by domain knowledge and Claude Opus co-authoring. Effective but risky — no spec to validate against.

Calibration Sprint (Mar 10): Spec-first. The superpowers methodology was applied: docs/superpowers/specs/2026-03-10-accuracy-calibration-design.md was written before docs/superpowers/plans/2026-03-10-accuracy-calibration.md, which preceded implementation. Tests followed immediately. This was the most disciplined development phase.

Phase 2 Planning (Mar 26–28): PRD and roadmap written after Phase 1 was complete. Planning served Phase 2 scoping, not Phase 1 execution.

Tests: Unit tests appeared on Mar 10 (with calibration), e2e tests on Mar 26. Tests were written after code for the initial build, and with code during calibration.

What Worked

  • Velocity: Full CRUD app with domain-specific calculators (BOM, labor, conduit fill, NEC/TIA compliance) built and deployed in one day (Mar 2). 17 feature commits in 6 hours.
  • Claude Opus co-authoring: 37 co-authored commits spanning schema design, domain logic, UI, and deployment. Consistent commit message quality (conventional commits format).
  • Spec-first calibration (Mar 10): Design spec before plan before code before tests. Achieved +/-5% accuracy against a real-world estimate (Highlark Hotel). The superpowers methodology worked when applied.
  • Refine plugin: 7 automated cycles with quantified scoring drove design quality from 5 to 9. The cycle-6 pivot to Fluent design language shows the plugin can make non-trivial design decisions.
  • Two-contributor model: Clean separation of concerns between primary dev (core features) and second contributor (UI polish, e2e tests, PRD). PR-based integration.
  • Domain depth: BICSI-aligned labor rates, NEC conduit fill calculations, 38 real SKUs, 13 BOM categories. Not a toy — built with industry-accurate domain knowledge.

What to Improve

  • Add CLAUDE.md: Document architecture decisions, coding conventions, domain glossary (drops, runs, BOM categories), and testing strategy. Every AI session should start with full context.
  • Test-first for new features: The calibration sprint proved spec → plan → code → test works. Apply this to all Phase 2 features, not just accuracy work.
  • CI pipeline: Add GitHub Actions with vitest run and playwright test gates before merge. The test infrastructure exists but is not enforced.
  • Error monitoring: Add Sentry or Vercel's built-in error tracking before Phase 2 goes live. The logger exists but has no external sink.
  • Phase 1 had no spec: Write a brief architecture decision record (ADR) retroactively. For Phase 2, apply the superpowers spec-first pattern to each theme.

Recommendations for Phase 2

  1. Create CLAUDE.md in repo root with project context, domain glossary, and conventions before starting Phase 2 development.
  2. Apply superpowers spec-first to all 4 themes: Each theme in docs/prd/features/ should get a docs/superpowers/specs/ design doc before implementation begins.
  3. Set up CI: GitHub Actions running npm run test and npm run build on every PR. Add Playwright to a scheduled nightly run if too slow for PR checks.
  4. Continue the refine plugin: The design score progression (5→9) was the most measurable quality improvement in the project. Use it for each new UI feature.
  5. Add error monitoring: Vercel Monitoring or Sentry. The $120K Phase 2 proposal demands production-grade observability.
  6. Formalize the two-contributor workflow: PR #1 worked well. Define branch naming, review expectations, and merge criteria for Phase 2 collaboration.
